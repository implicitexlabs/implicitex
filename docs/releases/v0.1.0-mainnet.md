## Fee Model

- Fee is calculated as: `fee = (amount * feeBps) / 10000`
- `amount` = recipient receives exactly this value
- `totalDebit` = `amount + fee` — full debit from sender
- At 1%: sender sends 1.01 USDC, recipient receives 1.00 USDC, treasury receives 0.01 USDC
- Fee is routed by the contract at execution time — not held or reviewed manually

## Security Hardening Applied

OpenZeppelin v5 modules: `Ownable2Step`, `Pausable`, `ReentrancyGuard`, `SafeERC20`

Constructor guards:
- Rejects `treasuryAddress == address(this)`
- Rejects `initialMinTransfer == 0`

`transferWithFee` recipient guards:
- Rejects `address(0)`
- Rejects `address(this)`
- Rejects the configured USDC token contract address
- Treasury recipient is explicitly allowed by policy

`setTreasury`:
- Rejects `address(this)` as new treasury

`rescueERC20`:
- Owner-only
- Cannot rescue the configured USDC token
- Emits `TokensRescued`

Test suite: 45 Hardhat tests passing at hardened deployment.

### UX caveats at launch

- Hardened deployment verified but not yet live-enabled in frontend config
- Single-user prototype validated only — hardened deployment still needs controlled transfer smoke before public exposure
- Multi-path testing (insufficient funds, rejected approval, wallet disconnect mid-flow, network switch) not yet completed
- Mobile wallet testing (MetaMask mobile, WalletConnect) not yet completed
- Allowance visibility: users cannot see existing approval scope or revoke from within the UI
- Polygonscan source verification completed post-deployment (same session)
- Amoy testnet deployment not completed (faucet gas never arrived)
